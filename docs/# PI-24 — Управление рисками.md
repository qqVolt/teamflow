# PI-24 — Управление рисками

## Реестр рисков (Risk Register)

| # | Риск | Вероятность | Ущерб | Уровень | Стратегия | Реализация в коде |
|---|---|---|---|---|---|---|
| R-1 | Потеря данных при сбое сервера | Средняя | Высокий | 🔴 Высокий | Резервное копирование; в перспективе — MongoDB | `InMemoryStorage` → заменяется persistence-слоем |
| R-2 | Назначение задачи несуществующему пользователю | Высокая | Средний | 🟡 Средний | Валидация assignee перед сохранением | `TaskService.createTask()` проверяет `storage.getUser()` |
| R-3 | Некорректные данные от клиента | Высокая | Средний | 🟡 Средний | Входная валидация в моделях | `Task.validate()`, `User.validate()` |
| R-4 | Пропуск дедлайна задачи | Средняя | Высокий | 🔴 Высокий | Упреждающие уведомления за 24 ч | `Task.isDeadlineApproaching()`, `TaskService.getApproachingDeadlines()` |
| R-5 | Несанкционированный доступ | Низкая | Критичный | 🟡 Средний | Система ролей и прав | `User.hasPermission()`, `ROLE_PERMISSIONS` |
| R-6 | Непредвиденная исключительная ситуация в сервисе | Средняя | Средний | 🟡 Средний | `try/catch` во всех методах сервисов | `TaskService`, `UserService` — все методы обёрнуты |

## Примеры обработки рисков в коде

R-3: Некорректные данные
```js
// TaskService.createTask() — проверка перед сохранением
const task = TaskFactory.create(type, data);
const validation = task.validate();
if (!validation.valid) {
  return { success: false, errors: validation.errors };  // не сохраняем
}


R-2: Несуществующий assignee
if (data.assignee) {
  const assignee = storage.getUser(data.assignee);
  if (!assignee) {
    return { success: false, errors: [`Исполнитель ${data.assignee} не найден`] };
  }
}


R-4: Дедлайн
// Уведомление за 24 часа
const approaching = TaskService.getApproachingDeadlines(24);
approaching.data.forEach((t) => notifyUser(t.getAssignee(), t));
